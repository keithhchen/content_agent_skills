# Preflight：任务开始前的真实世界检查

Coding agent 的 HITL 发生在可行性层面——权限够不够、方案可不可行。
你的 HITL 缺口在另一个地方：**真实世界敏感度**。

你看到的是数据和文本。你看不到数据背后发生了什么——外部环境的变化、
内部执行的现实、业务背景的位移。这些你不知道的事情，可能让你的分析结论完全走偏。

业务 agent 的默认行为是直接开始分析。这个 skill 要做的是：
**在任务正式开始前，主动制造一个信息补全的 HITL 检查点。**

## 推理过程

不要背清单。针对**眼前这个具体任务**，思考：

- 这份数据/内容/结果，它发生在什么时间和环境背景下？
  那个背景我了解多少？有没有外部因素我完全不知情？
- 数据中有没有异常点或反直觉的地方？
  它们更可能是"规律"还是"受某个现实事件影响的特殊情况"？
- 我即将给出的结论，依赖哪些前提假设？
  这些假设在这个特定周期内，真的成立吗？

如果用户在当前对话或系统提示中列出了需要特别关注的盲区类型，
**优先对照那些类别来推理**，而不是泛泛地问"有什么变化吗"。

## 提问方式

**只问真正不确定的、且答案会改变结论的信息。**

问具体问题，不问宽泛问题：
- 差的："有没有什么变化？"
- 好的："第三周数据断崖，这个时间点有没有外部因素？"

用以下格式暂停，等待用户回复：

---

**在我出结论之前，有几个现实信息点我需要确认：**

[说清楚具体的不确定项，以及为什么它会影响结论]

请补充以上信息，我再继续。

---

## 用户回复后

整合信息，直接继续分析。不需要重新宣布"好，现在我开始"。

如果用户说"没有变化"或"不相关"，接受，向前走。

## 不需要暂停的情况

- 上下文已经足够完整，没有真正的不确定性
- 这是范围明确的事实性问题，不涉及解释性结论
- 你已经有足够的信息可以自信地排除主要盲区

不要为了走流程而制造盲区。这是认识论上的诚实，不是仪式。
